# Distributing AnyMail MCP

The distribution model is **direct download from a website + GitHub Releases**,
the way Cursor, Rectangle, or Ollama ship — *not* the Mac App Store. MAS requires
the App Sandbox, which forbids the two things this app does by design: spawn a
`node`/engine child process and write other apps' MCP config files. So MAS is out;
everything below is the notarized-download path.

This is the **complete spec + runbook**, ordered by what unblocks what. It ships two
things from one codebase: the **menu-bar app** (a signed DMG) and the **CLI/engine**
(npm / Homebrew / standalone binary). The one item that makes "download and run"
actually true for a non-technical user is [step 1](#1-self-contained-engine-the-real-blocker-) — do it first.

---

## Who does what

Every step below is tagged so it's clear what needs a human with an account/credential
versus what is code and tooling:

| Tag | Meaning |
|-----|---------|
| 🧑 **Maintainer** | Needs an Apple/npm/domain account, a payment, or a design decision. Cannot be scripted or done by a coding agent. |
| 🤖 **Automatable** | In-repo code, scripts, or CI. A coding agent can build and verify it without any external account. |
| 🤝 **Mixed** | The script/workflow is automatable, but it can only *run* once the maintainer supplies a secret (cert, API key, npm token). |

> **Read this first if you're the maintainer:** the only things that genuinely
> require *you* are the accounts and secrets in
> [Prerequisites](#prerequisites-maintainer-must-provide-) plus the design/publish
> decisions flagged 🧑 inline. Everything else is buildable ahead of time and just
> waits on those. None of it is needed for personal use — a self-built, unsigned
> app already runs on your own Mac (see [`app/BUILD.md`](app/BUILD.md)); signing +
> notarization only matter for *distributing to other people*.

## Prerequisites (maintainer must provide) 🧑

Gather these once. The first two block every signed/notarized path; the rest unblock
one distribution channel each. Nothing here can be produced inside a coding-agent
environment — they're tied to your identity and payment.

| Prerequisite | For | Notes |
|---|---|---|
| **Apple Developer Program** — ~$99/yr | Signing the app + DMG | Enroll at <https://developer.apple.com/programs/>. You must be the Account Holder (or Admin) to issue a Developer ID cert. |
| **"Developer ID Application" certificate** | Signing | Create a CSR in *Keychain Access → Certificate Assistant*, upload at *developer.apple.com → Certificates*, download + install. **The private key stays in your login Keychain** — it never leaves your Mac and can't be regenerated by anyone else. |
| **notarytool credentials** | Notarization | Either an **App Store Connect API key** (`AuthKey_XXXX.p8` + Key ID + Issuer ID) — recommended for CI — or an **app-specific password** for your Apple ID. |
| **npm account** (+ 2FA / automation token) | Publishing the CLI to npm | Free. `npm login`. For CI publishing, mint an **automation token** (`npm token create --read-only=false`). |
| **A place to host** the download page + appcast | Download page, Sparkle updates | GitHub Pages is enough and free. A custom domain is optional (nicer download URL). |
| **App icon** (design decision) | App bundle + DMG art | You choose the mark; wiring it in is 🤖 (see [step 3](#3-the-dmg--app-icon)). |
| **Sparkle EdDSA signing key** | Auto-update integrity | Generated with Sparkle's `generate_keys` (🤖 can run it), but the **private key is a secret you keep** — losing it breaks the update chain. |

## 1. Self-contained engine (the real blocker) 🚨 🤖

Today the menu-bar app runs the user's **system `node dist/index.js`** (see
`app/GmailMCP/NodeLocator.swift` → `EnginePaths.entry`, which currently falls back to
the `~/loki-labs/gmail-mcp` dev checkout). A downloaded app cannot assume Node is
installed — that's the same "it doesn't just work" trap as an unsigned app hitting
Gatekeeper. The engine must ship *inside* the `.app`.

Two ways, pick one:

- [ ] **Option A — compile a single binary (recommended).** Produce a standalone
      `anymail-engine` executable and bundle it at `Resources/engine/anymail-engine`.
  - Bun: `bun build ./src/index.ts --compile --outfile anymail-engine`
  - or **Node SEA** (`--experimental-sea-config`, Node 20+ single executable app).
  - (Not `pkg` — it's archived/unmaintained.)
- [ ] **Option B — bundle Node + `dist/` + `node_modules`** into `Resources/engine/`
      and point `enginePath` there. Simpler, ~40 MB heavier, no compile step, and it
      keeps `EngineSupervisor` spawning `node <entry> --http` unchanged (you bundle a
      copy of the `node` binary and point `NodeLocator` at it).
- [ ] **⚠️ Native addon caveat (the step most likely to bite).** `@napi-rs/keyring`
      is a prebuilt `.node` binary. A compiled bundle (Option A) usually can't inline
      it — copy the matching `keyring.darwin-{arm64,x64}.node` next to the binary and
      confirm it loads at runtime on a clean Mac. This also interacts with signing —
      see the library-validation note in [step 2](#2-code-signing--notarization-required-to-open-without-warnings).
- [ ] Update `NodeLocator` / `EnginePaths` in the Swift app to **prefer the bundled
      engine** over a system Node (the `EnginePaths.entry` lookup already has a
      `Resources/engine/...` slot first — extend it for the compiled-binary layout,
      and for Option B add a bundled-`node` candidate to `NodeLocator`).
- [ ] Decide **arch coverage**: Apple-Silicon-only (`arm64`) is simplest; a universal
      build (`arm64` + `x86_64`) covers Intel Macs but doubles the addon/binary work.
- [ ] **Acceptance:** on a Mac with **no Node and no Homebrew** (a fresh user account
      works), the app launches, the engine starts, and `add`/`test`/`search` succeed.

## 2. Code signing + notarization (required to open without warnings) 🤝

The maintainer provides the cert + notarytool key ([Prerequisites](#prerequisites-maintainer-must-provide-));
the signing/notarizing pipeline itself is a script (🤖) that reads them from env, so it
can be written and dry-run now and only *executed* once the secrets exist.

**The app is intentionally not sandboxed** (`app/GmailMCP/GmailMCP.entitlements`) and
**Hardened Runtime is already on** (`ENABLE_HARDENED_RUNTIME: YES` in `project.yml`) —
notarization *requires* Hardened Runtime, so that's correct as-is.

### 2a. Get the signing identity 🧑
- [ ] Enroll in the Apple Developer Program and issue a **Developer ID Application**
      certificate (see Prerequisites). Confirm it's installed:
      `security find-identity -v -p codesigning` should list `Developer ID Application: <NAME> (<TEAMID>)`.
- [ ] Create the **notarytool** credentials (API key recommended).

### 2b. Sign correctly — inside-out, no `--deep` 🤖
`--deep` is **wrong for signing** (Apple says don't use it to *apply* signatures) and
would fail notarization. Every nested Mach-O must be signed **individually, inner
first, app last**, each with a **secure timestamp** (`--timestamp`) and Hardened
Runtime (`--options runtime`):

```bash
ID="Developer ID Application: <NAME> (<TEAMID>)"
APP="AnyMail MCP.app"

# 1) Sign the native addon (and re-sign it under your Developer ID — see note below)
codesign --force --timestamp --options runtime --sign "$ID" \
  "$APP/Contents/Resources/engine/keyring.darwin-arm64.node"

# 2) Sign the engine binary WITH ITS OWN ENTITLEMENTS (see 2c)
codesign --force --timestamp --options runtime --entitlements engine.entitlements \
  --sign "$ID" "$APP/Contents/Resources/engine/anymail-engine"
#    (Option B: sign the bundled `node` binary here instead.)

# 3) …sign any other nested executables (Sparkle's XPC/Autoupdate helpers) here…

# 4) Sign the app bundle LAST, with the APP entitlements
codesign --force --timestamp --options runtime \
  --entitlements "app/GmailMCP/GmailMCP.entitlements" --sign "$ID" "$APP"

# Verify (here --deep IS allowed — it's only banned for *signing*)
codesign --verify --deep --strict --verbose=2 "$APP"
```

### 2c. Hardened-Runtime entitlements for the JS runtime — on the engine, not the app 🤖
This is the deepest gotcha. Under Hardened Runtime, a **JIT** runtime (V8 in Node/Bun)
crashes without explicit entitlements, and loading a **third-party `.node`** trips
library validation. These entitlements govern the **child process by its own
signature** — putting them on the app does nothing for the engine. So step 1 needs a
**second entitlements file**, `engine.entitlements`, applied to the engine binary in
2b step 2:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>com.apple.security.cs.allow-jit</key><true/>
  <key>com.apple.security.cs.allow-unsigned-executable-memory</key><true/>
  <!-- Needed ONLY if you do NOT re-sign the .node addon under your Developer ID
       (2b step 1 re-signs it, which is cleaner and lets you drop this line): -->
  <key>com.apple.security.cs.disable-library-validation</key><true/>
</dict>
</plist>
```

The app's own `GmailMCP.entitlements` stays as-is (sandbox off). Do **not** copy these
JIT keys onto the app — they belong to the runtime binary.

### 2d. Notarize + staple + verify 🤝
- [ ] Build the DMG first ([step 3](#3-the-dmg--app-icon)), then notarize the **DMG**
      (it contains the already-signed app):
      ```bash
      xcrun notarytool submit "AnyMail MCP.dmg" \
        --key AuthKey_XXXX.p8 --key-id <KEYID> --issuer <ISSUER-UUID> --wait
      xcrun stapler staple "AnyMail MCP.dmg"
      ```
- [ ] **Acceptance:** `spctl -a -vvv -t install "AnyMail MCP.dmg"` and
      `spctl -a -vvv "AnyMail MCP.app"` both report **accepted / Notarized Developer ID**,
      and the app opens with a double-click on a Mac that has never seen it.

## 3. The DMG + app icon

### 3a. DMG (custom branded background) 🎨 🤖 — *tooling already in this repo*
- [ ] `brew install create-dmg librsvg` (librsvg rasterizes the SVG background).
- [ ] Background asset lives at **`assets/dmg-background.svg`**; the script exports it
      to `@1x` (600×400) and `@2x` (1200×800) PNGs.
- [ ] Build the DMG with **`scripts/make-dmg.sh "AnyMail MCP.app"`**: it lays out the
      app icon on the left, a **/Applications** drop-link on the right, an arrow
      between, the custom background, and the app icon as the volume icon. Pass
      `DEVELOPER_ID=...` to sign the DMG too.

```
┌───────────────────────────────────────────────┐
│                AnyMail  MCP                     │
│                                                 │
│     ┌────┐                        ┌────┐        │
│     │📧  │   ──────drag──────▶    │📁  │        │
│     └────┘                        └────┘        │
│    AnyMail MCP                  Applications     │
└───────────────────────────────────────────────┘
```

### 3b. App icon 🧑 design → 🤖 wiring
- [ ] **Decide the mark** (maintainer). Produce a 1024×1024 master.
- [ ] **Wire it in** (automatable): add an `Assets.xcassets` with an `AppIcon` set to
      the Xcode target (add it to `sources` in `project.yml`), and generate
      `AppIcon.icns` so `make-dmg.sh` can use it as the DMG volume icon (it reads
      `Contents/Resources/AppIcon.icns`, which doesn't exist yet — there's no asset
      catalog in the project today).

## 4. Auto-update (Sparkle) 🤖 code + 🧑 key custody
- [ ] Add **Sparkle** (SwiftPM) to the target; publish an `appcast.xml` with
      **EdDSA-signed** updates so the app updates itself instead of the user
      re-downloading.
- [ ] Generate the EdDSA keypair with Sparkle's `generate_keys` — the **private key is
      a secret the maintainer keeps** (see Prerequisites); the public key goes in
      `Info.plist`.
- [ ] Host `appcast.xml` + the DMGs on GitHub Releases (or the download-page host).
- [ ] Remember Sparkle adds nested helper executables — they must be signed in
      [step 2b](#2b-sign-correctly--inside-out-no---deep-).

## 5. Homebrew (optional, popular) 🤝
- [ ] **Cask** for the app → `brew install --cask anymail-mcp` (points at the notarized
      DMG on the latest GitHub Release). Ship via your own **tap repo**
      (`MarcinWalendowski/homebrew-anymail`) first; submit to `homebrew-cask` once
      there's adoption. Creating the tap repo is a maintainer action (it's your GitHub
      account); the cask file itself is 🤖.
- [ ] **Formula** for the CLI (below).

## 6. CLI distribution (the no-GUI path)
The engine is also the CLI — advanced users don't need the app at all.

### 6a. npm 🤝 — *package needs fixes before it's publishable*
The current `package.json` would publish a **broken** package. Fix first (🤖):
- [ ] Add a **`"files": ["dist"]`** allowlist. Without it npm falls back to
      `.gitignore`, which **excludes `dist/`** — so `npm publish` today would ship a
      package with **no `dist/index.js`**, and the `bin` would point at nothing.
- [ ] Flip **`"private": false`**.
- [ ] The `bin` entry (`dist/index.js`) already has the required
      `#!/usr/bin/env node` shebang — verified — so `npx`/global install will work.
- [ ] The name **`anymail-mcp` is free** on npm (verified). Add a `prepublishOnly`
      that runs `npm run build` so `dist/` is fresh.
- [ ] Then (maintainer): `npm publish` → users run `npx anymail-mcp add you@gmail.com`
      or `npm i -g anymail-mcp`. CI can publish with an automation token.

### 6b. Standalone CLI binary 🤖
- [ ] The same compiled binary from [step 1](#1-self-contained-engine-the-real-blocker-),
      attached to each GitHub Release for people without Node.

### 6c. One-line installer 🤖
- [ ] `curl -fsSL https://<host>/install.sh | sh` — downloads the right binary for the
      arch (arm64/x64), verifies a checksum, drops it on `PATH`.

### 6d. Homebrew formula 🤝
- [ ] A formula in the tap ([step 5](#5-homebrew-optional-popular-)) for
      `brew install anymail-mcp` (CLI).

## 7. Download page 🤖
- [ ] A minimal landing page (GitHub Pages is enough): a **"Download for macOS"**
      button linking to the latest release DMG, the `npx` / `brew` one-liners for the
      CLI, and the App-Password setup note. Detect Apple-Silicon vs Intel if shipping
      both. The only 🧑 part is choosing/enabling the host (and a custom domain, if any).

## 8. Release automation (CI) 🤝
- [ ] A **macOS GitHub Actions** workflow, triggered on `v*` tags, that: builds the
      engine binary, builds + signs the app (inside-out, per [step 2b](#2b-sign-correctly--inside-out-no---deep-)),
      makes + signs the DMG, notarizes + staples, publishes to npm, and uploads the DMG
      + CLI binary to the GitHub Release. The workflow is 🤖; it needs **repo secrets
      the maintainer provides** (🧑): base64 **Developer ID** cert + its password, the
      **notarytool** API key (`.p8` + key id + issuer), the **npm** automation token,
      and the **Sparkle** EdDSA private key. (This can't run in a coding-agent
      environment — no Apple account, no signing secrets.)

## Sequencing — what unblocks what

```
Prereqs (🧑 accounts/keys) ─┐
                            ├─► can be gathered in parallel with all build work
Step 1 engine (🤖) ─────────┴─► Step 2 sign+notarize (🤝) ─► Step 3 DMG (🤖/🧑 icon)
     │                                                            │
     └─► Step 6b/6c CLI binary (🤖)        Step 4 Sparkle (🤖) ◄──┘
                                                                  │
Step 6a npm fixes (🤖) ─► npm publish (🧑) ─► Step 5 Homebrew (🤝)
                                                                  │
                              Step 7 page (🤖) ─► Step 8 CI ties it all together (🤝)
```

- **Do first, in parallel:** the maintainer starts [Prerequisites](#prerequisites-maintainer-must-provide-)
  (Apple enrollment can take a day) while the automatable build work (steps 1, 6a, 3a,
  7) proceeds — none of it needs the accounts.
- **First unblock:** step 1 (self-contained engine) makes every download path real and
  gates steps 2–5.
- **Cheapest shippable slice:** steps 6a + 6b — publishing the CLI to npm and a raw
  binary needs *no Apple account at all*. That can go out before the signed app.
